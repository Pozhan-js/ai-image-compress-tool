# AI 图片压缩工具 — 技术方案

## 1 概述

本文档为“AI 图片压缩工具”基于 JavaScript 技术栈的技术方案。前端采用 React + Vite，后端采用 Node.js，图片的压缩与处理由后端实现。目标是满足需求文档中的核心功能（批量压缩、压缩前后对比、文件管理、本地处理等），并兼顾性能、可扩展性与跨平台分发。

## 2 技术选型概览

- 前端：React + Vite（TypeScript 可选）
- 后端：Node.js（建议 LTS 版本）
- 图片处理核心库：`sharp`（基于 libvips）为主，必要时使用 `imagemin` 系列插件或其他原生工具（mozjpeg、pngquant、cwebp、svgo）做补充
- 队列与并发：基于 `worker_threads` 或 `bullmq`（配合 Redis）用于批量任务和伸缩
- 持久化：本地文件系统为默认（临时目录 + 输出目录），可选 S3 / 对象存储用于分发/云端备份
- 桌面打包：Electron（或 Tauri）将前端与后端打包为桌面应用；也可提供服务端部署选项
- 日志与监控：Winston / Pino（应用日志），Prometheus + Grafana（性能监控，可选）

## 3 系统架构

- 客户端（React + Vite）
  - 文件上传（拖拽/选择/文件夹）
  - 参数配置与预设
  - 压缩前后对比 UI（左右对比、滑动条、缩略图）
  - 任务列表与进度展示

- 后端（Node.js）
  - 接收上传文件接口
  - 压缩任务调度与执行（本地进程/线程池）
  - 存储管理（临时区→输出区）
  - 日志、审计与历史记录

- 可选：消息队列（Redis + Bull/BullMQ）用于任务持久化与重试

拓扑图（简述）
- 前端 ←→ 后端 API ←→ 本地文件系统/对象存储
- 后端内部：API 层 → 调度层（队列）→ 处理层（Worker/Thread）→ 压缩库（sharp 等）

## 4 API 设计（概要）

- `POST /api/upload` — 上传图片（单张或多张）
  - 请求：multipart/form-data
  - 返回：上传文件 ID 列表、元数据

- `POST /api/compress` — 创建压缩任务
  - 请求：task 配置（文件 ID 列表、模式/参数、输出格式、尺寸规则）
  - 返回：taskId

- `GET /api/task/:id/status` — 获取任务状态与进度

- `GET /api/task/:id/result` — 列出输出文件与统计信息（大小对比、耗时）

- `GET /api/file/:id/download` — 下载单个输出文件

- `GET /api/history` — 获取压缩历史记录

注：响应应包含明确的错误码与可重试语义，支持范围请求用于大文件下载。

## 5 后端压缩模块设计

核心职责：验证 → 转码/缩放 → 压缩 → 元数据处理 → 写回输出

1. 验证层
  - 文件格式、大小、数量上限检查
  - 安全检查（文件扩展名 vs MIME，防止恶意文件）

2. 转码与尺寸调整
  - 使用 `sharp` 进行缩放与格式转换（保留/删除 EXIF 可配置）
  - 支持按比例缩放或固定分辨率

3. 压缩策略
  - 预设：`high-quality`、`balanced`、`high-compress`
  - 无损与有损选项
  - 对不同格式选择合适的编码器：JPEG（mozjpeg）、PNG（pngquant）、WebP（libwebp）、AVIF（libavif / libvips 支持）
  - 智能模式：基于目标大小/感知质量（可选引入 SSIM/PSNR 估算或轻量 ML 模型进行质量判断）

4. 并发与资源控制
  - CPU 密集型任务使用 `worker_threads` 或子进程池，限制并发数防止 OOM
  - 可选使用 Redis + BullMQ 做任务排队与重试，支持优先级

5. 缓存与临时文件管理
  - 上传后写入临时目录，任务成功后移动到输出目录或压缩包
  - 自动清理策略（基于时间或磁盘阈值）

6. 安全与隐私
  - 所有处理均在本地进行（不默认上报或上传）
  - 对敏感元数据（EXIF）提供移除选项

## 6 前端设计要点

- UI 框架：React + Vite，建议使用 TypeScript
- 组件：上传区、缩略图列表、参数面板、对比展示（左右/滑动/闪烁）、任务列表
- 状态管理：React Context 或轻量状态管理库（如 Zustand）
- 文件预览：使用本地 Object URL 或 FileReader，避免将文件先上传到服务器以便快速预览（如果选择本地模式，可走 Electron 的本地 FS 访问）
- 国际化与主题：支持中/英、多主题（深色/浅色）

## 7 桌面打包与可分发策略

- 两种部署模式：
  1. 本地打包（Electron/Tauri）：前端打包嵌入，Node 后端作为内嵌服务或直接在主进程中调用压缩逻辑；优点：完全本地、无需安装依赖；缺点：体积较大
  2. 客户端-服务端模式：前端作为独立 UI，与本地或远程 Node 服务交互；优点：服务端可扩展为云服务

- 打包建议：Electron + `electron-builder` 创建跨平台安装包；为 Apple Silicon 编译独立二进制

## 9 性能与容量规划

- 单机并发：基于 CPU 核数与内存设置 worker 池大小（经验值：并发 worker ≈ CPU 核数的 0.5-1 倍，视内存和 libvips 使用情况调整）
- 大文件策略：大于一定阈值（如 10MB）采取流式处理，避免一次性读入内存
- 批量队列：采用队列与限速避免磁盘抖动
- 性能指标：平均单张处理时长、最大并发吞吐量、内存占用和磁盘 I/O；在 CI 或基准测试中采集

## 10 测试与质量保障

- 单元测试：服务逻辑、压缩参数计算
- 集成测试：API 上传/压缩/下载流程（可使用本地临时目录）
- 压缩质量回归：对同一组样本计算文件大小、SSIM/PSNR（可选）以验证质量阈值
- 端到端测试：桌面版本（Electron）或 Web 客户端 + API

## 11 CI/CD 与发布

- CI：GitHub Actions
  - 步骤：安装依赖 → 静态检查（ESLint/TypeScript）→ 单元/集成测试 → 构建工件
- CD：
  - Web 客户端：静态资源构建并发布到 CDN/GitHub Pages
  - Electron：使用 `electron-builder` 在 CI 中打包并上传发行资产

## 12 安全与隐私

- 本地优先：默认不上传图片到第三方服务器
- 输入校验：严格校验上传文件类型与大小
- 最小权限：文件读写仅限工作目录与用户选择目录
- 元数据处理：提供移除 EXIF 的选项以保护隐私

## 13 可扩展性与后续方向

- 插件化压缩器：抽象压缩适配层，支持引入新的编码器或云端服务
- ML 驱动的质量评估：后续可加入轻量感知模型优化质量/大小平衡
- 多平台本地优化：针对 Apple Silicon、Windows ARM 做编译与性能优化

## 14 交付物与时间节点（建议）

- 里程碑 1（2 周）：基础前端上传 UI + 简单后端压缩 API（sharp）
- 里程碑 2（2 周）：批量任务、队列与进度、压缩预设
- 里程碑 3（2 周）：对比视图、历史记录、桌面打包 PoC
- 里程碑 4（1-2 周）：测试覆盖、性能调优、发布打包

## 15 结论与建议

本技术方案以 JavaScript 生态为基础，结合 `sharp`（libvips）作为核心压缩引擎，采用 React + Vite 提供高效的前端体验，并通过 Node.js 后端实现性能可控且易部署的压缩服务。对桌面场景建议使用 Electron 将前端与后端打包为本地应用以保证本地处理与隐私需求；同时保留云端部署选项以适配团队与企业用户。后续可通过队列、并行池与编解码器扩展进一步提升吞吐与兼容性。

## 开发环境 — 桌面应用调试启动

- 说明：开发时优先采用桌面应用（Electron）模式启动，便于访问本地 Node 压缩服务和文件系统，同时保证处理在本地完成以满足隐私要求。

- 建议启动流程：

1. 根目录安装依赖：

```bash
# 使用 pnpm/npm/yarn 等包管理器
pnpm install
```

2. 启动后端 API 服务（开发模式）：

```bash
# 在 packages/api-server
pnpm --filter @app/api-server dev
# 或
cd packages/api-server && pnpm dev
```

3. 启动前端（Vite）开发服务器：

```bash
# 在 apps/web-client
pnpm --filter web-client dev
# 或
cd apps/web-client && pnpm dev
```

4. 启动 Electron（渲染进程指向 Vite dev server）：

```bash
# 在 apps/desktop-app
pnpm --filter desktop-app dev
# dev 脚本应完成：
# - 将 RENDERER_URL 指向 Vite 的 dev 地址（例如 http://localhost:5173）
# - 启动 electron 主进程并加载远程 URL 以便热重载
```

- 推荐在根 `package.json` 添加组合脚本以便同时启动三者：

```json
"scripts": {
  "dev:api": "pnpm --filter @app/api-server dev",
  "dev:web": "pnpm --filter web-client dev",
  "dev:electron": "pnpm --filter desktop-app dev",
  "dev": "concurrently \"pnpm run dev:api\" \"pnpm run dev:web\" \"pnpm run dev:electron\""
}
```

- 调试要点：
  - 主进程调试：可用 `electron --inspect` 启动并在 VSCode 中通过 `attach` 配置附加调试器。
  - 渲染进程调试：在 Vite 提供的浏览器开发者工具中调试渲染逻辑；在 Electron 中可打开开发者工具进行联合调试。
  - IPC 与权限：渲染进程不直接访问底层文件系统。通过主进程提供受控的 IPC 接口（白名单方法）来执行文件读写与压缩任务请求，避免安全问题。
  - 环境配置：使用 `.env.development` 或 `process.env` 注入 `DEV=true`、`RENDERER_URL`、`API_HOST` 等变量，确保在开发/生产之间切换友好。
  - 本地数据：上传文件写入临时目录以便本地调试；记得清理临时文件以免占满磁盘。
